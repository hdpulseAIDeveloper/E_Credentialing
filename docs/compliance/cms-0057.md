# CMS-0057-F Compliance (Provider Directory)

CMS-0057-F ("Advancing Interoperability and Improving Prior Authorization Processes") requires payers, and by extension credentialing organizations, to publish provider directory information via FHIR R4 APIs.

## Platform coverage

The platform exposes a full **FHIR R4 Provider Directory** conforming to the
DaVinci PDex Plan-Net Implementation Guide. The following resource types
are advertised in the CapabilityStatement at `/api/fhir/metadata`:

| Resource | Surface | Wave |
| --- | --- | --- |
| `Practitioner` | search + read + `$everything` operation | initial + Wave 3.3 |
| `PractitionerRole` | search + read | initial |
| `Organization` | search + read | initial |
| `Location` | search + read | initial |
| `Endpoint` | search + read | initial |
| `HealthcareService` | search + read (derived from `DirectoryPractitionerRole` tuples) | **Wave 3.3** |
| `InsurancePlan` | search + read (derived from `Enrollment.payerName` where status is `ENROLLED`) | **Wave 3.3** |

Cross-cutting:

- API-key authentication on every request (scope `fhir:read`); errors emitted as `OperationOutcome`.
- Standard `_count`/`_offset` paging with `self`/`next`/`previous` Bundle links.
- Rate limiting (per-key fixed window) and full audit logging via `auditFhir`.
- `$everything` operation on `Practitioner` returns a single Bundle containing
  the practitioner + every related directory resource (roles, orgs, locations,
  endpoints, derived HealthcareServices, derived InsurancePlans). This is the
  canonical CMS-0057-F surveyor flow: one URL, one Bundle, every piece of
  evidence in one shot.

See [`docs/api/fhir.md`](../api/fhir.md) for the per-endpoint reference and
[`docs/qa/per-flow/fhir-everything.md`](../qa/per-flow/fhir-everything.md) for
the `$everything` per-flow card.

## Data included

For each approved provider:

- NPI (`identifier` with system `http://hl7.org/fhir/sid/us-npi`).
- Internal platform id (`identifier` with system `http://essenhealthcare.com/providerId`).
- Official name (legal).
- Work email and business phone (telecom).
- Active licenses with issuer and expiration (`qualification`).
- Board certifications (`qualification`).
- `active` flag reflects current credentialed status.

## Data excluded

PHI is never published through FHIR:

- No SSN
- No date of birth
- No home address
- No home phone
- No document content

## Currency

- Updates propagate within 1 minute of staff changes.
- `_lastUpdated` filter supports incremental sync.
- Full refresh pulls are supported but rate-limited; use incremental sync where possible.

## Roadmap

The full PDex Plan-Net resource set is now implemented. Remaining roadmap items:

- Modeled `DirectoryHealthcareService` and `DirectoryInsurancePlan` Prisma
  tables (Wave 5.x) so customers can override the derived fields with
  bespoke values without giving up the discovery contract.
- `Patient/$everything` and consent-driven access (only relevant if the
  CVO is later asked to publish patient-facing endpoints).
- SMART-on-FHIR app launch (deferred — current consumers all use API-key
  auth).

## Conformance testing

We run automated contract tests against the FHIR endpoint using the NCQA-published test vectors and HL7's Inferno validator. Failures block merge.

See [`docs/dev/testing.md`](../dev/testing.md).
