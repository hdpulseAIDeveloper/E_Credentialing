# DEF-0010 — Marketing homepage `/` renders 200 without a `<main>` landmark (PRE-EXISTING, surfaced by Pillar S)

## Metadata

- **DEF ID:**            DEF-0010
- **Opened:**            2026-04-19 by Cursor (Composer), surfaced by the first run of `npm run qa:live-stack`
- **Severity:**          medium (a11y violation; per STANDARD.md §4 axe `serious`/`critical` would catch downstream, but the structural landmark gap also fails Pillar S §5)
- **Pillar (A–S):**      A — Functional smoke (page-shape invariant), E — Accessibility (WCAG 2.1 SC 1.3.1, 2.4.1 — landmarks), **S — Live-Stack Reality Gate** (the surface that found this)
- **Affected route(s):** `/`
- **Affected role(s):**  anonymous (public marketing page)
- **Browser:**           all
- **Spec(s) failing:**   `scripts/qa/live-stack-smoke.mjs` Surface 5
- **Status:**            **CLOSED 2026-04-19** — fixed in the same commit as DEF-0008/DEF-0011 (the post-Pillar-S "first batch of gate-surfaced defects" cleanup commit)

## Captured evidence (REQUIRED — per STANDARD.md §4.1 step 1)

First run of `npm run qa:live-stack` against the post-DEF-0009 stack:

```
  FAIL [S.5.public] /  — 200 but blank-shell shape (<main>=false, <h1|h2>=true)
```

The page returns 200 and has at least one `<h1>` or `<h2>`, so it is not a
DEF-0003 / DEF-0004 hard-blank-shell — but it is missing the `<main>`
landmark, which:

1. Fails the Pillar S §5 page-shape invariant.
2. Is a WCAG 2.1 SC 1.3.1 / 2.4.1 violation that axe-core would
   classify as `moderate`. Pillar E's anonymous matrix would catch it
   when it runs.
3. Means screen-reader users cannot use the "skip to main content"
   shortcut on the marketing page — the very page most likely to be
   the first impression for new prospects.

## Root cause (hypothesis — not yet verified)

The marketing landing page at `src/app/page.tsx` likely wraps content in
a `<div>` rather than a `<main>` element, OR its layout
(`src/app/layout.tsx`) does not provide a `<main>` for the `(public)`
route group while the `(staff)` route group's layout does.

## Fix shape (recommended)

One of:

1. Wrap the top-level container in `src/app/page.tsx` in `<main>`.
2. If the public layout already has a `<main>`, remove the duplicated
   wrapper from `page.tsx`.

The fix is a 1-line change. It is being filed as a separate card
rather than fixed inline because the Pillar S landing commit is
explicitly scoped to the standard + the gate, not to the bugs the
gate finds (per STANDARD.md §4.1.1 "one root cause per commit", and
to keep the structural change reviewable on its own).

## Fix

Wrapped the three content sections in `src/app/page.tsx` (the hero,
the features grid, and the stats strip) in a single `<main id="main">`
element, kept between `<nav>` and `<footer>` per WAI-ARIA Landmarks.
The `<nav>` and `<footer>` correctly remain OUTSIDE `<main>` so the
"skip to main content" shortcut lands a screen-reader at the hero
heading, not at the brand link.

Verification (post-fix `npm run qa:live-stack`):

```
PASS [S.5.public] /  — 200 + page shape ok
```

Pillar S §5 page-shape invariant green for `/`. Pillar E axe-core
matrix re-runs on the next CI cycle and is expected to lose the
WCAG 2.1 SC 1.3.1 / 2.4.1 violation that this card documented.
