## Root cause (three stacked, all real, all required to reach green)

1. **Stale named-volume contents shadowed the freshly-rebuilt image.**
   `docker-compose.dev.yml` lines 56–62 declare named volumes
   `ecred_web_node_modules` and `ecred_web_next_cache` that survive
   container recreation by design (so a `docker compose up` does not
   wipe `node_modules` and force a multi-minute reinstall every time).
   When the image was rebuilt with the latest `master` carrying the
   new multi-tenant Prisma client, the named volumes still held the
   prior `node_modules/.prisma/client/` AND the prior
   `.next/server/...` compiled `auth.ts` — both of which were authored
   against a schema with `User.organizationId`. The runtime read the
   stale code path before the fresh image's `node_modules` mounted.

2. **Three Prisma migrations had never been applied to the live database.**
   Specifically: `20260418000000_add_telehealth_expirable_types`,
   `20260418100000_multitenancy_shim`,
   `20260418130000_billing_subscription_state`. Even after the volume
   cleanup made the fresh `User.organizationId` Prisma client
   load-bearing, the database still had no `organization_id` column.
   `prisma migrate status` would have shouted; nothing was running it.

3. **Dev Dockerfiles copied `prisma/` AFTER `npm install`.**
   `Dockerfile.web` and `Dockerfile.worker` were both written before
   the `postinstall: prisma generate` hook was added to
   `package.json`. Once that hook landed, every cold rebuild of the
   dev images died at `npm install --legacy-peer-deps` with
   `Could not find Prisma Schema`. The prod Dockerfiles were already
   correct (they're rebuilt rarely and were last touched after the
   hook landed). The dev pair was missed because the named-volume
   shadow let warm rebuilds keep working.

## Why the existing pillars did not catch it (the structural miss)

This is the part of the defect card that the user explicitly asked
us to address: **"What is the point of the multi-hours spent to test
if the first thing fails."** The honest answer is that none of the
gates as wired were probing the running system.

| Pillar  | Wired into qa:gate? | Would have caught it? | Why it didn't |
|---------|---------------------|----------------------|---------------|
| A — Smoke | No (only via `npm run qa:smoke`) | Yes — `globalSetup` runs the exact CSRF + credentials sign-in matrix that failed | Pillar A was honestly reported "Not Run" because Docker was not available at the time; per STANDARD.md §3 that should have failed the gate, but the headline was treated as informational |
| B — RBAC | No | Yes — same `globalSetup` dependency | Same |
| J — Contract | Yes | No — exercises the OpenAPI shape, not auth state |  |
| M — Data integrity / migrations | No | Could have — but no script wraps `prisma migrate status` | No `qa:migrations` existed |
| Q — Documentation integrity | Yes | No — links + ADR sanity, not runtime |  |
| `qa:gate` itself | (this is the gate) | No — runs ZERO specs; only `qa:inventory && qa:cards:check && qa:coverage && sdk:check && postman:check` | The gate was never asked to probe the running system |

## Fix (operational + structural)

### Operational (already applied in the prior loop, 2026-04-19)

1. `docker compose -f docker-compose.dev.yml down ecred-web`
2. `docker volume rm e_credentialing_ecred_web_node_modules e_credentialing_ecred_web_next_cache`
3. `docker compose -f docker-compose.dev.yml up -d ecred-web`
4. `docker compose -f docker-compose.dev.yml exec ecred-web npx prisma migrate deploy`
5. Sign in with the canonical seeded credentials from `prisma/seed.ts`:
   `admin@hdpulseai.com` / `Users1!@#$%^`. Result: 302 → `/dashboard`,
   `authjs.session-token` cookie set, `/api/auth/session` returns
   `{ user: { role: "ADMIN", email: "admin@hdpulseai.com", ... } }`.

### Structural (this commit — closes DEF-0009 permanently)

The dev Dockerfiles were already fixed in commit `e9c0fc2` (copy
`prisma/` BEFORE `npm install --legacy-peer-deps`). The remaining
work in THIS commit:

- **STANDARD.md bumped 1.1.0 → 1.2.0**. New Pillar S — Live-Stack
  Reality Gate added (§2.S). New §4 hard-fail conditions (11)–(14)
  for migration drift, dead seed accounts, cold Dockerfile build,
  named-volume staleness. New §10.2 names this regression.
- **`scripts/qa/live-stack-smoke.mjs`** (HTTP-only, no browser
  required). Probes `/api/health`, runs the role-by-role CSRF
  sign-in matrix from `tests/e2e/roles.ts`, an authenticated
  `/dashboard` shape probe, every public-route shape probe, and
  optionally introspects the running container's prisma client to
  detect named-volume staleness.
- **`scripts/qa/check-migration-drift.mjs`** wraps
  `prisma migrate status`, fails on any pending / drift / connection
  failure (per ADR 0028 §Anti-weakening 3, no-connection ≠ no-pending).
- **`scripts/qa/check-dockerfile-build.mjs`** lints every compose
  file with `config -q`; in `--cold` mode rebuilds every app service
  with `--no-cache`.
- **`tests/e2e/live-stack/role-login-matrix.spec.ts`** — browser-driven
  supplement that catches hydration / JS-execution regressions on the
  sign-in page.
- **`npm run qa:gate`** rewritten to include `qa:migrations` and
  `qa:live-stack`. The static-only path is preserved as
  `qa:gate:static` for the lint-only CI shard.
- **ADR 0028** captures the decision, the four alternatives that
  were rejected, and the eight anti-weakening rules.
- **Definition of Done** updated with three new boxes (#11 wire-up,
  #12 trigger files, #13 schema-field migration parity).
