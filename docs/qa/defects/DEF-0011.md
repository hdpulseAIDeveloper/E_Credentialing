## Captured evidence (REQUIRED — per STANDARD.md §4.1 step 1)

First run of `npm run qa:live-stack` against the post-DEF-0009 stack:

```
  FAIL [S.5.public] /settings/billing  — expected 200, got 307 (Location: /auth/signin?callbackUrl=...)
  FAIL [S.5.public] /settings/compliance  — expected 200, got 307 (Location: /auth/signin?callbackUrl=...)
```

The Pillar S smoke iterates `docs/qa/inventories/route-inventory.json`
filtering `group === "public"`. Both `/settings/billing` and
`/settings/compliance` appear in that filter, but the runtime
middleware (`src/middleware.ts`) redirects them to `/auth/signin`
because they are — by every reasonable product reading — staff-only
internal admin pages. The `/settings/*` namespace is, on every other
production-like CVO, an authenticated-only surface.

## Root cause (two possibilities the user must pick between)

### Hypothesis A — inventory is wrong (likely)

`scripts/qa/build-route-inventory.ts` (or the page's frontmatter /
exports) is mis-classifying these as `public`. Both pages should be
`group: "staff"`. Fix: re-run `npm run qa:inventory` after correcting
whatever signal feeds the classifier; commit the regenerated
`route-inventory.json`.

### Hypothesis B — middleware is wrong (unlikely)

The pages were intended to be public-readable (e.g. a public-facing
billing-status page or a compliance-disclosure page) and the
middleware's blanket `/settings/*` redirect is the bug. Fix: extend
the middleware allow-list to include `/settings/billing` and
`/settings/compliance` (note: this is the same shape as DEF-0008's
remaining work).

## Why this matters

This is the second class of structural bug Pillar S is designed to
surface (alongside DEF-0008): **drift between the inventory's
declared access posture and the runtime middleware's enforced access
posture**. Both DEF-0008 (middleware behind inventory) and DEF-0011
(inventory ahead of middleware OR inventory wrong) are now
deterministically reproduced every gate run.

The right structural answer remains the one DEF-0008 escalated for:
either generate the middleware allow-list from the inventory at build
time (`scripts/qa/build-public-allowlist.ts`), or move every
`group:public` page under a `(public)/` route group so middleware can
match on segment groups instead of an explicit list.

## Fix (2026-04-19)

Hypothesis A confirmed — `/settings/billing` and `/settings/compliance`
are staff-only admin pages by product intent (per their `page.tsx`
contents and per the `(staff)/settings/...` pattern that the rest
of the staff settings tree uses).

The bug was in `scripts/qa/build-route-inventory.ts`: any page whose
on-disk path did not contain a `(group)/` segment fell through the
classifier and defaulted to `group: "public"`. Since `/settings/billing`
and `/settings/compliance` live directly under `src/app/settings/...`
(not inside `src/app/(staff)/settings/...`), they hit the default
branch and were silently mis-classified.

The structural fix is shared with DEF-0008: a new
`src/lib/public-routes.ts` exports `isPublicRoute(pathname)` (the
SAME predicate the middleware uses). The inventory builder now
classifies in this order:

1. Explicit `(group)/` segment — wins (authorial intent).
2. Otherwise, `isPublicRoute(route)` → `public`.
3. Otherwise, `isProviderPortalRoute(route)` → `provider`.
4. Otherwise, **`staff`** — the truthful default, since the runtime
   middleware redirects everything not in the public allow-list to
   `/auth/signin`.

After regenerating: `/settings/billing` and `/settings/compliance`
correctly classify as `staff`. Verified by reading
`docs/qa/inventories/route-inventory.json` post-fix:

```
SETTINGS:
 /settings/billing -> staff
 /settings/compliance -> staff
```

The `public` set now contains exactly the 16 routes the middleware
allows — the inventory and the runtime middleware can no longer
disagree because they share `src/lib/public-routes.ts`.
