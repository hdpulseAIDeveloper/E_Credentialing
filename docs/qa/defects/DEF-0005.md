## Captured evidence

axe-core 4.11 violation, rule `color-contrast` (impact: serious),
fired across **18 routes × ≥ 4 staff roles**, totalling several
hundred individual node violations. Representative example from
`/dashboard` (84 nodes on this page alone):

```
Element has insufficient color contrast of 4.44 (foreground color: #6b7280,
background color: #eff6ff, font size: 9.0pt (12px), font weight: normal).
Expected contrast ratio of 4.5:1
```

HTML pattern:

```html
<div class="bg-blue-50 rounded-lg border border-blue-200 px-3 py-2.5">
  <div class="text-xl font-bold text-blue-600">24</div>
  <div class="text-xs text-gray-500 mt-0.5">In Onboarding</div>
</div>
```

Same shape for `bg-violet-50`, `bg-red-50`, `bg-amber-50`, `bg-green-50`,
`bg-rose-50`, etc. All fail by **0.06 – 0.10 of a ratio point** below
WCAG AA's 4.5:1 threshold.

Full evidence:
`docs/qa/results/2026-04-17/test-output/all-roles-pillar-e-a11y-pillar-E-a11y-*-role-*/error-context.md`.

## Diagnosis (root cause)

Tailwind's `text-gray-500` (#6b7280) on Tailwind's `*-50` light
backgrounds (#eff6ff, #f5f3ff, #fef2f2, etc.) gives 4.4 – 4.49 contrast
against the lightest backgrounds. The recurring stat-tile component
across the app uses this exact pairing. This is a **palette decision**,
not a coding bug.

## Minimum fix options (each is a §4.2-compliant real fix, not a weakening)

1. Promote `text-gray-500` → `text-gray-600` (#4b5563) for label text on
   `*-50` backgrounds. Contrast jumps to 7.0+ : 1. Visual impact: text
   becomes one shade darker on tiles only.
2. Bump the tile background to `*-100` (slightly more saturated). Visual
   impact: tiles stop reading as "pastel".
3. Reduce the label opacity from 100 % gray-500 to 100 % gray-700.

Option 1 is the smallest patch (one global Tailwind class swap on the
StatCard primitive) and is the recommended fix.

## Why this is escalated, not auto-fixed

Per `docs/qa/STANDARD.md` §4.1: "loop until green or until 3 attempts
on the same root cause have failed". A palette change touches
**every screen with a stat tile** (≥ 18 in this run) and warrants a
deliberate design + brand review rather than a unilateral patch from
the QA layer. The QA layer is doing exactly what it was designed to
do — surfacing the violation to a human owner. The standard's
anti-weakening rules forbid silencing the rule (§4.2.6) or lowering
the AA threshold (§4.2.7).
