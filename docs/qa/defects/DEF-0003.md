| Field      | Value                                                  |

## Captured evidence

User-reported runtime error after credential login on `/dashboard`:

```
Warning: Expected server HTML to contain a matching <a> in <div>.
    at a
    at LinkComponent (next/link)
    at div
    at nav
    at div
    at Sidebar (src/components/layout/sidebar.tsx:93:11)
…
Uncaught Error: Hydration failed because the initial UI does not match
what was rendered on the server.
```

Followed by the cascade in DEF-0004 (`Cannot read properties of undefined
(reading 'call')` at the webpack factory).

## Diagnosis

Inspection of `src/components/layout/sidebar.tsx` showed clean Link/`<a>`
nesting (Next 14 default rendering, no `legacyBehavior`, no nested `<a>`).

The dev-server log captured by `docker logs ecred-web` showed:

```
⚠ Fast Refresh had to perform a full reload due to a runtime error.
```

The combination of "Fast Refresh full reload" + "factory undefined" +
"hydration mismatch" is the canonical signature of a **stale Next.js dev
build cache** after a structural source change (in this case, the legal
copy bundle added new client components and reorganised
`src/app/(provider)/layout.tsx`). The corrupted webpack module map served
a partial chunk, the parent tree fell back to client rendering, and the
client tree no longer matched the server-rendered HTML — producing the
hydration warning.

## Minimum fix

Stop the dev container, clear the named cache volume, restart:

```
docker compose -f docker-compose.dev.yml stop ecred-web
docker run --rm -v e_credentialing_ecred_web_next_cache:/cache alpine \
  sh -c "rm -rf /cache/* /cache/.* 2>/dev/null"
docker compose -f docker-compose.dev.yml start ecred-web
```

(For non-dockerised local runs: `rm -rf .next` then `npm run dev`.)
