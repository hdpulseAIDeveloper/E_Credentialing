## Root cause

Two separate, well-intentioned design decisions collided:

1. **Wave 11 (`scripts/qa/build-postman-collection.ts`, `2026-04-18`)**
   wrote the auto-generated collection to `public/api/v1/postman.json`.
   The header docstring's anti-weakening rule even said: _"The script
   writes to `public/api/v1/postman.json`, NOT to `docs/api/`. The
   contract is 'Next.js serves it' â€” the file is a build artifact, not
   source documentation."_ At Wave 11, the contract was: Next.js's
   built-in static file server serves the artifact directly from
   `public/`.

2. **Wave 17 (`src/app/api/v1/postman.json/route.ts`, `2026-04-18`)**
   added a custom App Router route handler at the same URL to layer
   on (a) Wave 17's ETag/If-None-Match conditional GET (saves
   bandwidth on repeated polls), (b) a `Content-Disposition: attachment;
   filename="ecredentialing-v1.postman_collection.json"` header so
   downloads land with a sensible filename in Postman/Insomnia/Bruno,
   and (c) explicit `Cache-Control` and `X-Content-Type-Options`
   hardening. The route handler still read from
   `public/api/v1/postman.json`.

The Wave 17 author did not notice that putting a route handler at
the same URL as a `public/` file is a Next.js error condition (the
docs at the linked page are explicit: "If you create a public file
and a page file with the same path, Next.js will throw an error").
The error wasn't visible in production because:

- `next dev` shows the error at request time (500 + diagnostic JSON),
  but local devs typically test the OpenAPI URLs (`/api/v1/openapi.{json,yaml}`)
  not the Postman one â€” and there was no contract test for the
  HTTP delivery surface (Pillar J's existing
  `tests/contract/pillar-j-postman.spec.ts` validates the
  on-disk JSON contents against the OpenAPI spec, NOT the served
  HTTP response).
- Production never hit the route either, becauseâ€¦ the Pillar J
  contract test gave a green light by virtue of the on-disk JSON
  being correct, and no monitor probed the live URL.

Pillar S Surface 5 was the first gate that actually fetched
`/api/v1/postman.json` over HTTP from the deployed stack â€” and
flipped red on the very first probe.

## Why this is a high-severity defect (not a "nice-to-have")

The Postman collection is the **first thing** a prospective API
customer downloads after reading the OpenAPI spec. The `/sandbox`
page links to it. The `/api/v1/openapi.json` contract description
(line 1 of the `info.description` field) advertises it. The README,
ADR 0023, and the Wave 11 changelog entry all promote it as the
v1 onboarding path. A 500 on first download is a credibility-
destroying first impression â€” it's the technical equivalent of
the demo crashing in the sales meeting.

## Fix landed (2026-04-19)

**Source artifact moved:** `public/api/v1/postman.json` â†’
`data/api/v1/postman.json`. The empty `public/api/v1/` and
`public/api/` directories were removed.

**Why `data/` and not somewhere under `src/`:** keeping it
co-located with the route (`src/app/api/v1/postman.json/postman.json`)
would work technically (only `route.ts`/`route.js` create routes
in App Router), but it muddles the conceptual model: the file is a
**build artifact** that the route handler **delivers**, not a
**source file** that the route handler **is**. A new top-level
`data/` directory makes that distinction explicit, sits as a
sibling to `public/`, and is forward-compatible with eventually
moving `docs/changelog/public.md` and `docs/api/openapi-v1.yaml`
into `data/` (the cleaner long-term home for runtime-read build
artifacts â€” see DEF-0012's "Option 2" recommendation).

**Files touched:**

- `src/app/api/v1/postman.json/route.ts` â€” `loadCollection()` now
  reads `join(process.cwd(), "data", "api", "v1", "postman.json")`.
  The route header docstring now explicitly explains why the
  source lives outside `public/` (with a link to the Next.js
  conflicting-public-file-page docs page) so future contributors
  don't accidentally reintroduce the bug.
- `scripts/qa/build-postman-collection.ts` â€” `OUT_PATH` updated;
  the anti-weakening rule expanded to forbid writing to `public/`.
- `scripts/qa/check-postman-drift.ts` â€” `COMMITTED` path updated;
  drift-fix instructions updated to reference the new location.
- `tests/contract/pillar-j-postman.spec.ts` â€” `COLLECTION_PATH`
  updated.
- `docker-compose.dev.yml` â€” added `- ./data:/app/data` bind
  mount for the dev container (with an inline comment naming
  this defect and the structural reason).
- `.dockerignore` â€” `data/` is intentionally NOT excluded, so
  the prod image picks up `data/api/v1/postman.json`
  automatically.

**Pillar S hardening:** `scripts/qa/live-stack-smoke.mjs` Surface 5
now probes `/api/v1/postman.json` on every gate run, validating
HTTP 200, `Content-Type: application/json`, and a body â‰Ą 5,000
bytes. A regression to either the conflict-class bug (500) or
the silently-missing-artifact class (200 with empty body) trips
Pillar S immediately.
