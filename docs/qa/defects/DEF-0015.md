## Root-cause analysis

`tsconfig.worker.json` declares `"include": ["src/workers/**/*",
"src/lib/**/*", "src/server/db.ts"]`. That blanket `src/lib/**/*`
sucked in **every** `src/lib/` file the web tier ever added — including
the Wave 5.3 file `src/lib/billing/stripe-client.ts`, which imports
`@/env`, which is `src/env.ts`, which uses `createEnv` from
`@t3-oss/env-nextjs`.

`@t3-oss/env-nextjs` is published with a modern package
`exports` map and only exposes its type declarations through that
map. The worker's `tsconfig.worker.json` was set to
`"moduleResolution": "node"` (the legacy "node10" algorithm), which
ignores `exports` maps and falls back to walking `node_modules`
directly. The legacy walker cannot find the declaration file inside
the `exports`-only layout, so `tsc` errors with

```
TS2307: Cannot find module '@t3-oss/env-nextjs' …
```

even though the file is physically present at
`/app/node_modules/@t3-oss/env-nextjs/dist/index.d.ts` (which `tsc`
helpfully points out in its own error message).

Why didn't this fire earlier?

1. The worker's compile graph used to consist of bot files,
   integration helpers, and audit/email/encryption utilities —
   none of which imported `src/env.ts`. So the `node` resolution mode
   never had to resolve a modern-`exports` package.
2. Wave 5.3 (`db17ecd`) added `src/lib/billing/stripe-client.ts`,
   which (correctly, for the web tier) imports `@/env`. The web's
   `tsconfig.json` uses Next.js's modern `moduleResolution: "bundler"`
   so the web build kept passing.
3. The worker's blanket `src/lib/**/*` include picked up the new
   billing file even though the worker has no use for it.
4. The cold-Dockerfile gate (`npm run qa:dockerfile:cold`) is
   gated behind `.env` presence for production compose files
   (per the `--skip-prod` graceful-degradation logic added in
   the Pillar S Wave) and reported NOTRUN locally. The CI matrix
   does run it but only on PRs that touch `Dockerfile.*`; the
   docs-only commit that immediately preceded this deploy did
   not trigger it. So the regression shipped silently, exactly
   as DEF-0009 did before Pillar S existed.

In short: the worker's compile contract drifted — it now claimed to
compile a file (`src/lib/billing/stripe-client.ts`) it had no business
compiling, in a TS resolution mode that could not handle modern
package layouts.

## Fix-Until-Green attempts

### Attempt 1 — 2026-04-19 19:10 by Cursor

- **Hypothesis:** the worker has no functional dependency on
  `src/lib/billing/**`, `src/lib/api/**`, `src/lib/changelog/**`,
  `src/lib/sandbox/**`, `src/lib/auditor/**`, `src/lib/ai/**`,
  `src/lib/legal/**`, or `src/lib/api-client/**`. Adding these to
  `tsconfig.worker.json` `exclude` will narrow the worker's compile
  graph back to its actual surface and stop pulling in `@/env`.
  This is preferable to changing `moduleResolution` because:
  (a) it is the single-source-of-truth-style fix (the worker's
      tsconfig should reflect what the worker actually compiles), and
  (b) it preserves the proven CJS output shape (`module: commonjs`,
      `node dist/workers/index.js`) without the chain of changes
      `node16`/`nodenext` would force on `module`, on tsc-alias
      output paths, and on every `.ts` file under `src/workers/**`.
- **Fix applied (commit SHA):** _to be assigned_ — surgical edit to
  `tsconfig.worker.json` adding the eight web-only subtrees to
  `exclude`.
- **Pillar(s) re-run:** S (cold Dockerfile build, locally), full
  `npm run typecheck`, full `npm run build:worker`.

## Forward-looking hardening (the structural part)

Closing this card operationally is not enough — the same regression
class will recur the next time someone adds a new `src/lib/<name>/`
subtree that imports a modern-`exports` package. Two structural
follow-ups are filed:

1. **Tighten Pillar S Surface 6 to never report NOTRUN silently
   for `docker-compose.prod.yml`** — currently the gate emits
   `NOTRUN` when `.env` is missing, which is the failure mode that
   let DEF-0015 ship. Either run the cold-build with a synthetic
   `.env.prod.fake` (env values are not exercised at build time —
   only at runtime), or escalate the NOTRUN to a hard fail when
   the run is part of a release-gate context (`--release`).
2. **Migrate `tsconfig.worker.json` to `moduleResolution: "node16"`
   + `module: "node16"` in a follow-up PR** so future modern-`exports`
   packages are resolved correctly even if the worker eventually
   needs to compile them. Out of scope for this defect because
   the immediate exclusion fix is necessary AND sufficient for
   green production today; the resolution-mode migration is a
   broader change that deserves its own PR with a full
   `npm run qa:gate` round-trip.

These two items are tracked here rather than spawning two new
defect cards because they are the *anti-weakening* of the present
fix; they MUST land within the next two PRs touching the worker
build or its compose file.
